Batch обновление данных

Обновление большого объема данных связано с некоторыми проблемами, основными из которых являются собственно сам большой объем данных и длительность транзакции. Для обновления или удаления большого объема данных лучше использовать bath метод выполняя операции над небольшими объемами данных.

Пример работы с batch-update

Создадим тестовую таблицу для дальнейшего обновления.

drop table if exists tbltestbachdata; create unlogged table tbltestbachdata(id int primary key not null generated by default as identity, name text, newvalue text); -- Заполняем тестовые данные insert into tbltestbachdata(name) select v::text ||'---'|| (random() * 100)::text as values from generate_series(1,500000) as v; select * from tbltestbachdata limit 10; select count(*) from tbltestbachdata;

Создадим тестовую таблицу-ключей, т.е. те данные которые мы хотели бы обновить

-- таблица ключей, которые нужно обновить drop table if exists keytbl; create unlogged table keytbl as select ROW_NUMBER() OVER(ORDER BY id) row_id, -- создаем собственный упорядоченный счетчик id from tbltestbachdata limit 5001; ------------------------------- create index on keytbl (row_id); select * from keytbl limit 10;

Обновление стандартным образом:

update tbltestbachdata s set newvalue=s.name || 'upd' from keytbl toup where toup.id = s.id;

План стандартного обновления:

Обновление с использованием batch-update

do $$ declare total_count int = (select count(*) from keytbl); batch_size int = 1000; iterations int = (total_count / batch_size); ---------------- start_id int = batch_size; end_id int = (1) * batch_size; begin raise notice 'total_count=% batch_size=% iterations=%',total_count, batch_size, iterations; for i in 0..iterations loop start_id = i * batch_size; end_id = (i+1) * batch_size; raise notice 'start_id=% end_id=% iterations=%',start_id, end_id, i; update tbltestbachdata s set newvalue=s.name || 'upd' from keytbl toup where toup.id=s.id and toup.row_id>start_id and toup.row_id<=end_id; --perform pg_sleep(1); end loop; end; $$;

План выполнения:

update tbltestbachdata s set newvalue=s.name || 'upd' from keytbl toup where toup.id=s.id and toup.row_id>start_id and toup.row_id<=end_id Update on public.tbltestbachdata s (cost=0.71..3314.31 rows=0 width=0) (actual time=17.323..17.324 rows=0 loops=1) Buffers: shared hit=13423 dirtied=8 written=8 -> Nested Loop (cost=0.71..3314.31 rows=999 width=44) (actual time=0.041..5.217 rows=1000 loops=1) Output: (s.name || 'upd'::text), s.ctid, toup.ctid Inner Unique: true Buffers: shared hit=4546 -> Index Scan using keytbl_row_id_idx on public.keytbl toup (cost=0.28..32.76 rows=999 width=10) (actual time=0.027..0.499 rows=1000 loops=1) Output: toup.ctid, toup.id Index Cond: ((toup.row_id > 0) AND (toup.row_id <= 1000)) Buffers: shared hit=10 -> Index Scan using tbltestbachdata_pkey on public.tbltestbachdata s (cost=0.43..3.28 rows=1 width=36) (actual time=0.003..0.003 rows=1 loops=1000) Output: s.name, s.ctid, s.id Index Cond: (s.id = toup.id) Buffers: shared hit=4536

Обращаем внимание на использование индексов:

  • Index Scan using keytbl_row_id_idx on public.keytbl

  • Index Scan using tbltestbachdata_pkey on public.tbltestbachdata

Время выполнения при использовании batch-update может быть больше, чем при использовании стандартного обновления. Но стоит обратить внимание на строку perform pg_sleep(1); - которая делает паузу между обновлениями, что дает промежуток времени другим операциям для работы с таблицей.

В данном случае у вас есть пространство для маневра - подобрать более оптимальное количество строк для обновления за один раз в соответствии с доступными ресурсами.
